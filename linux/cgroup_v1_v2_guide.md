# Linux cgroup과 cgroup v1, v2 차이

## 1. cgroup이란
cgroup은 Linux 커널의 **프로세스 그룹 단위 자원 관리 기능**이다.  
여러 프로세스를 하나의 그룹으로 묶고, 그 그룹에 대해 CPU, 메모리, I/O, PID 수 같은 자원을 제한하거나 사용량을 추적할 수 있게 해준다. Docker, containerd, Kubernetes, systemd 같은 구성 요소가 이 기능을 사용한다. 커널 문서도 cgroup을 그룹에 대한 자원 추적과 제한을 위한 메커니즘으로 설명한다.  

## 2. cgroup의 핵심 개념

### 2.1 cgroup
프로세스를 묶어 관리하는 단위다.

### 2.2 controller
특정 자원을 제어하는 커널 기능이다. 대표적으로 아래가 있다.
- cpu
- memory
- io
- pids
- cpuset

### 2.3 계층 구조
부모 cgroup 아래에 자식 cgroup을 두는 트리 구조로 자원을 분배한다.

### 2.4 파일 기반 인터페이스
보통 `/sys/fs/cgroup` 아래의 파일을 읽고 쓰는 방식으로 설정한다.  

## 3. cgroup을 왜 쓸까?
cgroup은 다음과 같은 상황에서 사용한다.
- 어떤 서비스가 메모리를 일정량 이상 쓰지 못하게 제한할 때
- 특정 프로세스 그룹의 CPU 사용량을 제한할 때
- 컨테이너별 PID 개수를 제한할 때
- 서비스별 자원 사용량을 추적할 때
- 시스템 서비스와 사용자 워크로드를 분리할 때

즉, cgroup은 **리소스 제한**, **격리**, **우선순위 조정**, **사용량 측정**을 위해 사용한다.  

## 4. cgroup v1
cgroup v1은 초기 세대 구현이다. 가장 큰 특징은 **컨트롤러별로 서로 다른 계층을 따로 가질 수 있다**는 점이다.
예를 들어 아래처럼 나눌 수 있다.

- cpu 전용 계층
- memory 전용 계층
- cpuset 전용 계층

즉, 같은 프로세스가 CPU 계층에서는 한 위치에 있고, memory 계층에서는 다른 위치에 있을 수 있다. 커널 문서도 v1에서 여러 active hierarchy를 동시에 둘 수 있다고 설명한다.  

### 4.1 v1의 장점

- 초기에 컨트롤러별 유연한 분리가 가능했다.
- 오래된 사용자 공간 도구와 호환성이 있다.

### 4.2 v1의 문제점

- 컨트롤러마다 트리 구조가 달라질 수 있다.
- 같은 프로세스가 자원 종류마다 다른 cgroup에 속할 수 있다.
- 관리가 복잡하다.
- 컨트롤러 간 동작 방식이 일관되지 않다.
- 계층형 자원 분배 규칙이 애매한 부분이 있다.

## 5. cgroup v2

cgroup v2는 v1의 구조적 문제를 정리하기 위해 나온 **통합 계층 방식**이다. 가장 큰 특징은 **모든 컨트롤러를 하나의 공통 계층에서 관리한다**는 점이다. 커널 문서도 v2를 unified hierarchy 기반 인터페이스로 설명한다.  

### 5.1 v2의 목표
- 구조를 단순하게 만든다.
- 컨트롤러 동작을 일관되게 만든다.
- 계층형 자원 분배를 명확하게 만든다.
- 사용자 공간 관리 도구가 다루기 쉽게 만든다.

### 5.2 v2의 특징

- 모든 컨트롤러가 하나의 트리 안에서 동작한다.
- 파일 이름과 인터페이스가 더 정리되어 있다.
- 부모와 자식 간 자원 분배 규칙이 더 엄격하다.
- 내부 프로세스를 가진 노드와 하위 cgroup을 동시에 혼합하는 방식에 제약이 있다.

이 구조는 자원 분배를 더 예측 가능하게 만든다.  

## 6. cgroup v1과 v2 차이

| 항목 | cgroup v1 | cgroup v2 |
|---|---|---|
| 계층 구조 | 컨트롤러별 별도 계층 가능 | 하나의 통합 계층 사용 |
| 관리 방식 | 복잡하다 | 단순하다 |
| 컨트롤러 일관성 | 낮다 | 높다 |
| 프로세스 배치 | 자원별로 다른 트리에 속할 수 있다 | 하나의 트리 기준으로 관리한다 |
| 인터페이스 | 컨트롤러별로 제각각이다 | 파일명과 규칙이 더 정리되어 있다 |
| 자원 분배 모델 | 느슨하거나 불일치한 부분이 있다 | 계층형 분배 규칙이 더 명확하다 |
| 현재 방향성 | 레거시 취급이다 | 최신 표준 방향이다 |

## 7. 실무적으로 보면 어떤 차이인가

v1은 기능이 먼저 생기고 관리 모델이 뒤따라온 구조에 가깝다. 그래서 자원별 제어는 가능하지만 전체 구조가 복잡하다.  
v2는 관리 일관성과 운영 편의성을 더 중요하게 반영한 구조다.

실무적으로는 아래처럼 이해하면 된다.

- v1은 자원별로 따로 관리하던 예전 방식
- v2는 하나의 트리에서 통합 관리하는 최신 방식

## 8. 왜 v2가 더 선호될까?
- 컨트롤러별 계층 분리로 인한 혼란을 줄일 수 있다.
- 시스템 전체 자원 관리 모델이 단순해진다.
- systemd와 최신 배포판이 v2 중심으로 맞춰지고 있다..
- 컨테이너 환경에서 동작을 더 예측 가능하게 볼 수 있다

systemd 문서도 cgroup 기반 자원 제어를 설명하면서 unified hierarchy를 기준으로 동작을 구분한다.
